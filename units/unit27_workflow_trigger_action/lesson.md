# [[Unit27]] lesson: ワークフロー設計：トリガーとアクション

## 学習ゴール
- Lv.3ワークフローを、何をきっかけに動くかというトリガーと、動いた後に何をするかというアクションの連鎖として設計する。

## 講義ノート
### scene01 トリガーとアクション
- ワークフロー設計は、何をきっかけに動くか、動いた後に何をするかを分けて考えると整理できます。
- トリガーは開始の合図
- アクションは実行される処理
- 用語: ワークフロー = 起点から複数処理を順番につなぐ自動化の設計図。

### scene02 三つのトリガー
- トリガーは大きく、指示型、定時型、条件型に分けられます。業務が始まる自然なきっかけを先に見るのがコツです。
- 指示型は人の意思
- 定時型は時計
- 条件型は変化の検知
- 用語: トリガー = ワークフローを起動するきっかけ。

### scene03 人が合図を出す
- 指示型は、人間の意図的な行為で起動します。チャット指示やボタン実行、フォーム送信などが代表例です。
- 人が今やってほしいと示す
- 会話だけで終わらず後続処理へつなぐ

### scene04 時計が合図を出す
- 定時型は、毎朝、毎週、期限の何日前といった時間で動きます。予測しやすく、導入初期にも扱いやすい起点です。
- 固定時刻ならスケジュール
- データ内の日付なら期限ベース
- 用語: 期限ベース = 契約更新日など、データに入った日付を基準に起動する方式。

### scene05 変化を検知して動く
- 条件型は、メール到着、ファイル追加、CRMステータス変更、閾値超過のような変化を検知して起動します。
- 状態変化を見つけて動く
- 見落としや遅延を減らせる
- 用語: イベントトリガー = システム内の状態変化を検知して起動する方式。

### scene06 Webhookとイベント
- イベントトリガーはワークフロー側が変化を見に行く考え方。Webhookは外部システム側が出来事を通知して起動します。
- 見る主体が違う
- 外部から通知されるならWebhook
- 用語: Webhook = 外部システムが出来事をワークフローへ通知する方式。

### scene07 三つのアクション
- アクションは、データ処理、生成・判断、実行・連携に分けると整理しやすい。インプット、思考、アウトプットの流れです。
- データを整える
- AIが生成・判断する
- 結果を書き出す
- 用語: アクション = トリガー後に実行される処理。

### scene08 取得・抽出・整形
- データ処理では、APIで情報を取り、文書から必要情報を抽出し、後続処理が扱いやすい形式に整えます。
- APIコールは外部から取得
- 抽出は手元データから取り出す
- 整形で形式をそろえる

### scene09 AIが考える部分
- 生成・判断では、返信文や提案書の下書きを作る、内容を分類する、人に回すか自動返信するかを判断します。
- コンテンツ生成
- 分類・振り分け
- 判断・条件分岐

### scene10 提案書AIへつなぐ
- 提案書AIでは、案件ステージ変更を起点に、顧客情報取得、過去提案書検索、ドラフト生成、通知、保存をつなぎます。
- 自然な起点はCRMの状態変化
- 複数アクションを順序でつなぐ
- 用語: 連鎖設計 = 一つのトリガーから複数処理を順番につなぐ設計。

## 試験での見分け方
- 具体例を見たら、先に判定軸を言語化してから選択肢を見る。
- ツール名だけで判断せず、起動条件、連携有無、計画主体、処理内容で判別する。
