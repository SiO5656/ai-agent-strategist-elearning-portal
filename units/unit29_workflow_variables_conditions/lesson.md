# [[Unit29]] lesson - 変数と条件分岐

## 学習ゴール
- ワークフローでなぜ変数と条件分岐が必要か説明できる。
- If/Else、ルーティング、フィルタリングの違いを見分けられる。
- 問い合わせ・提案ワークフローで、どの変数を使ってどの粒度で分岐させるか判断できる。
- 失敗しやすい分岐設計を避け、既定ルートやフォールバックを設計できる。

## 1. なぜ条件分岐が必要か
現実の業務では、すべての案件を同じ処理へ流すと品質が落ちる。顧客種別、緊急度、金額、状態、カテゴリが違えば、必要なテンプレート、担当者、承認、通知も変わる。条件分岐は、単純な直線のワークフローを、状況に合わせて動く業務フローへ変える仕組み。

## 2. 変数とは何か
変数は、ワークフロー内で値を保持する名前付きの箱。フォーム入力、メール送信者、CRMの顧客属性、在庫数、AIの分類結果、計算済みの合計金額などを持ち、後続ステップで再利用する。

例:
- `customer_type`: 新規 / 既存 / VIP
- `urgency`: 高 / 中 / 低
- `amount`: 金額
- `status`: 未対応 / 対応中 / 完了
- `category`: 技術 / 請求 / 営業 / その他

## 3. 条件分岐の3型
### If/Else
二択の分岐。条件を満たす場合と満たさない場合で処理を分ける。

例: 緊急度が高いなら担当者へ通知、そうでなければ自動返信。

### ルーティング
3方向以上の振り分け。複数カテゴリを一箇所で管理でき、入れ子のIf/Elseより見通しがよい。

例: 問い合わせ内容を技術、請求、営業、その他へ振り分ける。

### フィルタリング
通過か遮断かの選別。処理対象として残すか、対象外として除外するかを決める。

例: 受注済みの提案書だけを参照対象に残し、失注分は除外する。

## 4. 実務シナリオ
問い合わせ対応ワークフローでは、次のように分ける。

1. トリガーで問い合わせを受け取る。
2. 顧客種別、緊急度、状態、金額、カテゴリを変数として保持する。
3. カテゴリが3つ以上ならルーティングで担当領域へ振り分ける。
4. 緊急度が高いかどうかはIf/Elseで担当者通知を切り替える。
5. 対象外レコードや重複イベントはフィルタリングで早めに除外する。
6. 分岐後の返信、通知、承認、保留などはアクション設計として決める。

## 5. 設計の粒度
分岐は細かければ良いわけではない。すべての変数で分けると保守が難しくなる。逆に一切分けないと、顧客や案件に合わない処理になる。成否に効く重要変数へ絞り、既定ルートを必ず用意する。

## 6. 失敗パターン
- 分岐ルールが複雑すぎる。
- どの条件にも合わない時の既定ルートがない。
- 古い変数や更新前の状態を参照している。
- 条件名や判定基準があいまい。
- 異常時に人へ戻すフォールバックがない。

## 試験での見分け方
- 二択: If/Else。
- 3方向以上: ルーティング。
- 通過/遮断: フィルタリング。
- 名前付きの値を後続で使う: 変数。
- 分岐後に何を実行するか: アクション設計。
