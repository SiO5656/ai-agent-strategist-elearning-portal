# Unit14 講義ノート

## テーマ
[[BPR]]と[[ECRS]]：自動化前に業務を見直す

## 学習目標
- [[BPR]]を「既存業務の小さな改善」ではなく「業務プロセスの再設計」として説明できる。
- 「汚いプロセスの自動化」がなぜAIエージェント導入の失敗につながるかを説明できる。
- [[As-Is分析]] → [[ECRS]] → [[To-Be設計]] の順序を理解できる。
- [[ECRS]]の4観点、`Eliminate` / `Combine` / `Rearrange` / `Simplify` を改善案に対応づけられる。
- 現実の導入では、理想の業務再設計と現場の抵抗をどう合意形成するかを説明できる。

## 1. BPRは「改善」ではなく「再設計」
[[BPR]]は、既存の業務を少しずつ良くするだけではなく、「そもそもその業務や工程は必要か」を問い直す考え方です。

改善は、今ある流れを前提にして効率を上げます。再設計は、今ある流れが本当に必要かから見直します。AIエージェント導入では、この違いが大きな成果差につながります。

## 2. 汚いプロセスの自動化
AIエージェント導入で起こりやすい失敗は、非効率な業務をそのまま高速化してしまうことです。

たとえば、紙提出、複数承認、手入力、転記という経費精算フローを、そのままAIで自動化しても、不要な承認や紙前提の設計は残ります。速くなったように見えて、非効率が高速に回るだけです。

## 3. 正しい順序
重要な順序は次の通りです。

1. [[As-Is分析]]で現状業務を把握する。
2. [[BPR]] / [[ECRS]]で業務プロセスを見直す。
3. 最適化された業務に対して[[To-Be設計]]を行う。
4. 必要なところにAIエージェントを適用する。

## 4. ECRSの4観点
[[ECRS]]は、業務を見直すための実務フレームです。順序にも意味があります。

### Eliminate（排除）
「この工程は本当に必要か」を問います。不要なら丸ごとなくします。効果が最も大きく、排除できれば自動化も不要です。

### Combine（結合）
「分かれている工程をまとめられないか」を問います。工程間の待ち時間や引き渡しコストを減らします。

### Rearrange（順序変更）
「この順番である理由はあるか」を問います。順序を変えるだけで、手戻りや待ち時間が減ることがあります。

### Simplify（簡素化）
「もっと簡単にできないか」を問います。残った工程の入力項目、形式、選択肢、テンプレートを整理します。

## 5. ヘルプデスク業務への適用
[[Unit13]]で可視化したヘルプデスク業務では、ECRSを次のように使えます。

- Eliminate: FAQ回答への過剰な承認をなくす。
- Combine: カテゴリ分類とFAQ検索を同時に行う。
- Rearrange: 回答後にログを残すのではなく、問い合わせ受信時点でチケットを起票する。
- Simplify: 非FAQ案件の調査メモを自由記述からテンプレート入力にする。

## 6. 現実的な運用
BPRは強力ですが、全社を一気に白紙から作り直すと、現場の抵抗や混乱が大きくなります。

AIエージェント導入では、対象業務に絞って小さくECRSを適用し、理想のTo-Beと現実の制約を関係者と合意しておくことが重要です。

## まとめ
[[BPR]]と[[ECRS]]は、AIエージェントを入れる前に業務を整えるための考え方です。非効率な業務をそのまま自動化するのではなく、先に不要な工程をなくし、統合し、順序を見直し、簡素化する。この順序を守ることで、AIエージェントは単なる効率化ツールではなく、業務変革の起点になります。
