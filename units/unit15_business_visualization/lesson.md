# [[Unit15]] 業務可視化の4手法：IPOから業務フロー図まで

## 学習ゴール
- [[AIエージェント]]導入前に、なぜ業務を見える形に外部化する必要があるかを説明できる。
- [[IPO]] / [[SIPOC]] / [[HTA]] / [[業務フロー図]] の目的と粒度の違いを区別できる。
- 目的・対象者・詳細度に応じて、どの可視化手法を選ぶべきか判断できる。
- ヘルプデスク業務の例で、4手法を段階的に使う流れを説明できる。

## 1. なぜ業務を可視化するのか
[[AIエージェント]]に業務を任せる前に必要なのは、「業務の中身を正確に説明できる状態」を作ること。
長年続く業務ほど、担当者の経験・判断・例外処理が頭の中に閉じやすい。この状態は[[暗黙知]]であり、AIにも新人にも引き継ぎにくい。

業務可視化は、単なる記録ではなく、AIに渡す情報・判断・担当範囲を設計するための土台になる。

## 2. 4手法の全体像
粒度が粗い順に並べると次の通り。

1. [[IPO]]: 入力・処理・出力で業務の輪郭をつかむ。
2. [[SIPOC]]: 供給者・入力・処理・出力・顧客で関係者を含めて整理する。
3. [[HTA]]: 業務を階層的に分解し、最小作業単位で担当を決める。
4. [[業務フロー図]]: 担当者・部署・順序・引き渡しを1枚の図に統合する。

4つを毎回すべて使う必要はない。目的と粒度に合わせて選ぶ。

## 3. IPOモデル
[[IPO]]は、Input / Process / Output の3要素で業務を見る。

例: 受注処理
- Input: 発注書、在庫データ
- Process: 発注内容確認、在庫引当
- Output: 受注確認、出荷指示

AI設計では「AIに何を渡し、何を返してもらうか」を決める最初の整理になる。
ただし、誰が入力を用意し、誰が出力を受け取るかまでは見えにくい。

## 4. SIPOC分析
[[SIPOC]]は、Supplier / Input / Process / Output / Customer の5要素で整理する。
[[IPO]]だけでは見落としやすい関係者、特にOutputの受け取り手を洗い出せる。

例: 受注処理では、顧客や倉庫だけでなく、売上データを参照する経営企画部門もCustomerになる可能性がある。
ここを見落とすと、導入直前に追加要件が出て手戻りが起きる。

## 5. HTA
[[HTA]]は Hierarchical Task Analysis の略。業務を階層的に分解し、アトミック・タスク単位で担当を決める。

担当分類の基本:
- AIエージェント: ルールが明確で、繰り返し発生し、判断や読解を伴う作業。
- 定型処理 / RPA / スクリプト: 判断を伴わない機械的な操作。
- 人間: 例外判断、倫理的判断、高度な文脈理解、責任を伴う判断。

例:
- 添付PDFを取得する → 定型処理
- 発注書から品目・数量・納期を抽出する → AI処理
- 大口注文の受注可否を判断する → 人間

## 6. 業務フロー図
[[業務フロー図]]は4手法の中で最も詳細。担当者や部署をレーンに並べ、工程を矢印でつなぐ。[[スイムレーン図]]とも呼ぶ。

価値:
- 工程の抜けが見える。
- 担当の重複が見える。
- 引き渡しの曖昧さが見える。
- As-IsとTo-Beを比較し、現場の合意形成に使える。

## 7. ヘルプデスク業務での使い方
1. IPO: 問い合わせ、OS環境、FAQを入力。分類・FAQ判定・回答生成・エスカレーション判定を処理。回答、チケット、月次レポートを出力。
2. SIPOC: 社員、FAQ管理者、IT管理者、経営層などを関係者として洗い出す。
3. HTA: 問い合わせ分類はAI、メール送信は定型処理、クレーム方針は人間などに分ける。
4. 業務フロー図: 社員、AIエージェント、IT担当者、管理者をレーンに並べ、解決までの流れを共有する。

## 8. 他人の業務を聞き出す5つの問い
- 起点: 何がきっかけで始まるか。
- 実作業: 実際にどの順番で画面や手を動かすか。
- イレギュラー: 例外時に何を判断するか。
- 終わり: 完了条件と渡す成果物は何か。
- 関係者: 自分以外に誰が関わるか。休んだ時に誰が代わるか。

## 試験で問われやすい観点
- 「入力と出力を整理したい」なら[[IPO]]。
- 「出力を受け取る関係者を洗い出したい」なら[[SIPOC]]。
- 「AI・定型処理・人間の担当を分けたい」なら[[HTA]]。
- 「チームで流れと担当を共有したい」なら[[業務フロー図]]。
