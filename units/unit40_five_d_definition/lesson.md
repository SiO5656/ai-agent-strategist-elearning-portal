# [[Unit40]] 5Dモデル Step2：Definition — 定義・構造化

## 学習範囲
- source pages: 360-368

## 学習目標
- DefinitionはDiscoveryで選んだユースケースを設計・開発に渡せる具体的な形へ落とし込む段階である。
- 最初に業務フローチャートで暗黙知を可視化し、AIと人間の役割分担を切り分ける。
- 要件定義書は機能要件、非機能要件、制約条件、前提条件を整理する共通の土台である。
- ペルソナは属性ではなく、AI出力をどう使い、どんな判断を下すかで描く。
- 合格基準はPoCの合否判定に使う一回限りの定量基準で、KPIとは区別する。

## 講義ノート

### 1. Definitionは設計へ渡す翻訳
- 選んだ業務を具体的な設計材料に変える
- 曖昧なままDesignへ進むと手戻りが増える
- 用語: **Definition** — 選定したユースケースを、業務フロー・要件・基準として具体化するステップ。

### 2. 業務フローで見える化する
- 暗黙知をフローチャートにする
- AIと人間の役割分担を切り分ける
- 用語: **業務フローチャート** — 業務の手順と判断点を、順番に見える形へ整理した図。

### 3. Discoveryより細かく分解する
- Discoveryは問題発見まで
- Definitionは業務手順と分担まで具体化する
- 用語: **粒度** — 同じ業務を、どの細かさで見るかという分解の細かさ。

### 4. 要件定義は共通の土台
- 完璧な文書を一人で作ることではない
- 検討の土台を言語化することが役割
- 用語: **要件定義書** — 実現したいこと、品質水準、守る条件、前提を整理した文書。

### 5. 何をするか／どの水準か
- 機能要件はAIにさせたいこと
- 非機能要件は速度・安定性・再現性などの条件
- 用語: **非機能要件** — 機能そのものではなく、速度・安定性・可用性などの品質条件。

### 6. 守るルールと満たす条件
- 制約条件は守るべきルール
- 前提条件は設計の土台として満たす条件
- 用語: **制約条件** — 設計や運用で守らなければならない業務・法務・倫理上のルール。

### 7. 属性より出力の使い方を見る
- AI固有のペルソナは出力の使い方で描く
- 直接ユーザーだけでなく受取者も定義する
- 用語: **ペルソナ** — AIエージェントの出力を使う典型的な利用者像。

### 8. PoCの合否を先に決める
- PoC後に主観で判断しない
- 業務要求から逆算した定量基準を置く
- 用語: **合格基準** — PoCで業務に使える品質に達したかを判断する定量基準。

### 9. 一回限りと継続測定を分ける
- 合格基準はPoCの合否判定
- KPIは運用中の継続的な成果測定
- 用語: **KPI** — 運用中に成果や状態を継続的に測るための指標。

### 10. Definitionは曖昧さを減らす
- AIに任せる範囲と人間の役割を明確にする
- Designへ渡せる材料に整える
