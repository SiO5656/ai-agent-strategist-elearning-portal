# データの種類と特性：構造化・非構造化・メタデータ

## この講義でできるようになること

- データを構造化・非構造化・半構造化に分類できる。
- それぞれのデータに対して、LLM・API・既存システムのどれを使うべきか判断できる。
- メタデータがRAGやナレッジベースの検索精度に与える影響を説明できる。
- ヘルプデスクAIや営業AIの業務フローで、データの種類に応じた処理方法を選べる。

## 1. なぜデータの種類を区別するのか

AIエージェント設計では、「このデータを誰が処理するべきか」を決める必要があります。

売上集計や社員情報のように項目が決まったデータと、メール本文や議事録のように自然言語で書かれたデータでは、適した処理方法が違います。

- 構造化データ: APIやデータベース処理が得意。
- 非構造化データ: LLMが意味理解や要約で力を発揮する。
- 半構造化データ: LLMの出力と後続システムをつなぐ。

すべてをLLMに任せると、コストや信頼性の面で無駄が出ます。逆に、従来システムだけでは、メールや文書の意味理解が難しくなります。

## 2. 構造化データ

構造化データとは、行と列、項目、型が明確なデータです。

代表例:

- 受注履歴
- 社員情報
- 売上集計
- 会計データ
- 顧客マスタ

構造化データは、検索・集計・比較がしやすく、データベースやAPIの定型処理に向いています。

AIエージェントで社員のOS環境を使いたい場合は、全社員情報をLLMに読ませるのではなく、社員IDをキーにAPIで必要な情報だけを取得し、プロンプトへ追加するのが効率的です。

## 3. 非構造化データ

非構造化データとは、明確な表構造を持たないデータです。

代表例:

- メール本文
- チャットメッセージ
- 議事録
- マニュアル
- FAQ本文
- PDFや画像、音声、動画

従来システムは、文章の意味や文脈を理解することが苦手でした。LLMはこの領域に強く、問い合わせ意図の理解、要約、回答案作成、文書参照などに向いています。

## 4. 半構造化データ

半構造化データとは、完全な表ではないものの、タグやキーによって部分的に意味が整理されたデータです。

代表例:

- JSON
- XML
- Webhookの通知データ
- AIエージェントの出力仕様

LLMの回答をJSON形式にすると、後続のシステムが処理しやすくなります。

例:

```json
{
  "category": "問い合わせ種別",
  "answer": "回答案",
  "confidence": 0.92
}
```

このように、半構造化データは、LLMの判断結果をワークフローやCRMなどへ渡すインターフェースとして機能します。

## 5. データ種類ごとの処理の担い手

覚えるべき基本形は次の通りです。

- 構造化データ: APIや既存システムで取得・更新する。
- 非構造化データ: LLMで意味理解・要約・回答生成を行う。
- 半構造化データ: JSONなどでAIの判断結果を後続システムへ渡す。

この分担により、精度・速度・コストを最適化しやすくなります。

## 6. 業務フローではデータが混在する

ヘルプデスクAIを例にすると、複数のデータが同じフロー内に現れます。

- 問い合わせメール本文: 非構造化データ。LLMが意図を理解する。
- 社員の部署やOS情報: 構造化データ。APIで取得する。
- マニュアル本文: 非構造化データ。RAGで参照する。
- 回答結果: 半構造化データ。JSONでチケットシステムへ渡す。

設計者は、「どの場面で、どの種類のデータが発生し、どの仕組みが処理するか」を整理します。

## 7. メタデータ

メタデータとは、データについてのデータです。

マニュアル本文そのものがデータだとすると、以下はメタデータです。

- カテゴリ
- 対応OS
- 最終更新日
- 対象部署
- 優先度

メタデータは、RAGやナレッジベースの検索精度を大きく左右します。本文が正しくても、カテゴリや対象条件が付いていないと、AIエージェントが適切な文書を見つけられないことがあります。

## 8. 非構造化データの前処理

マニュアルやFAQをAIエージェントに参照させるには、前処理が必要です。

- 長い文書を意味のまとまりへ分割する。
- 各まとまりにカテゴリや対象条件を付ける。
- FAQは質問と回答のペアで整理する。
- 表記ゆれを減らす。

長い文書を意味のまとまりへ分割する処理をチャンキングと呼びます。チャンクの大きさやメタデータ設計は、RAGの検索精度に影響します。

## 9. 今日から作るデータをAIが読みやすくする

過去データをすべて整備し直すのは現実的でない場合があります。そこで、今後作るデータをAIが読みやすい形で蓄積する運用が重要です。

実践例:

- 1つのセルには1つの情報だけを書く。
- 表記を統一する。
- ファイル名に日付や分類を含める。
- 文書にカテゴリ情報を付ける。
- FAQは質問と回答を分けて管理する。

データ整備は、AI導入の前準備ではなく、AI導入そのものの一部です。

## 10. 試験対策の要点

試験では、データ分類と処理方法の組み合わせが問われます。

典型パターン:

1. 自由記述の文章をLLMが理解・要約する。
2. LLMの出力をJSON形式に整える。
3. APIでCRMやチケットシステムへ登録する。

この流れは、非構造化データ → LLM → 半構造化データ → API → 構造化システム、という設計パターンです。
