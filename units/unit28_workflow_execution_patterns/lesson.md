# [[Unit28]] lesson: 実行・連携パターンと理解度チェック

## 学習ゴール
- Section20の理解度チェックでトリガーとアクション種別を固め、試験範囲外コラムとしてLv.2〜Lv.4の代表ツールと自律型エージェントの実態を整理する。

## 講義ノート
### scene01 Unit28の位置づけ
- Unit28は、トリガーとアクションの判別を問題で固め、さらに構築ツールと自律型エージェントの実態を参考知識として整理します。
- 前半は理解度チェック
- 後半は試験範囲外コラム
- 用語: コラム = 試験範囲外だが実務理解を補強する参考知識。

### scene02 イベントトリガー判別
- Salesforceの案件ステージ変更を検知して起動するなら、イベントトリガーです。人が変更していても、起点は状態変化です。
- CRMの状態変化を検知
- フォーム送信とは区別する
- 用語: イベント = システム内で起きた状態変化。

### scene03 APIコールとデータ抽出
- 外部サービスから情報を取るのがAPIコール。取得済みの情報から必要な項目を取り出すのがデータ抽出です。
- 取得はAPIコール
- 取り出しはデータ抽出

### scene04 トリガー三択の見分け
- 毎週金曜ならスケジュール。営業担当者がフォームを送るならフォーム送信。新規顧客登録を検知するならイベントです。
- 時刻固定はスケジュール
- 送信行為はフォーム
- 登録検知はイベント

### scene05 Lv.2ツールの選び方
- Lv.2のカスタムチャットボットは、自社のプラットフォーム環境との親和性を最優先で選びます。品質の中心はプロンプトとナレッジ設計です。
- 環境との親和性を優先
- 品質はコンテキスト設計で決まる
- 用語: ナレッジ設計 = AIが参照する文書や知識を用途に合う形で整えること。

### scene06 Lv.2の限界
- Lv.2は基本的に、人が質問してAIが答える受動型です。外部システムへの書き込みや承認処理が必要ならLv.3を検討します。
- 会話支援には強い
- 業務フロー実行には限界がある

### scene07 Lv.3ツール三分類
- Lv.3のツールは、エコシステム型、コネクタ型、開発型で整理できます。自社環境と処理の複雑さで選びます。
- エコシステム型
- コネクタ型
- 開発型
- 用語: iPaaS = 複数SaaSをつなぐ連携基盤。

### scene08 ツール併用の考え方
- 実務では、RAG付き処理本体は開発型、Salesforce検知とSlack通知はコネクタ型のように、役割で併用することもあります。
- 一つのツールに閉じない
- 処理本体と接続役を分ける

### scene09 自律型の実態
- Lv.4では、目標からToDoを作り、進行に合わせて更新し、エラーも学習材料として残します。停止ではなく回復が重要です。
- ToDoで目標を見失わない
- エラーから回復する
- 用語: エラー回復 = 失敗の痕跡を使い、次の行動を修正する能力。

### scene10 次Unitへの橋渡し
- 自律性が上がるほど、LLMに渡す情報設計が重要になります。次は、変数と条件分岐でワークフローを柔軟に制御する方法へ進みます。
- 判別軸を問題で固める
- ツール名より設計思想を覚える
- 次は条件分岐へ
- 用語: コンテキスト設計 = LLMに渡す情報を目的に合わせて選び、整える設計。

## 試験での見分け方
- 具体例を見たら、先に判定軸を言語化してから選択肢を見る。
- ツール名だけで判断せず、起動条件、連携有無、計画主体、処理内容で判別する。
