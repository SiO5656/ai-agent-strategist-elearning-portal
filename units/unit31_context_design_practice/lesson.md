# [[Unit31]] コンテキスト設計の実践：情報選択と品質改善

## このUnitでできるようになること
- AIに渡す情報を「必要かどうか」で選べる。
- 現在の事実、承認済み知識、類似事例、制約の優先順位を説明できる。
- ノイズ、矛盾、古い情報、資料内の紛らわしい指示を取り除ける。
- 顧客プロフィール、制約、評価基準、過去出力、例を構造化して渡せる。
- 出力を評価し、コンテキストを足す・削る改善ループを回せる。

## 1. コンテキスト設計は「選ぶ」仕事
コンテキスト設計で最初に考えるのは、情報を増やすことではなく、今回のタスクに必要な情報を選ぶことです。

「関係しそう」ではなく、「この情報がないと正しく判断できないか」で見ます。情報が不足するとAIは一般論に寄り、情報が多すぎると重要な前提が埋もれます。

## 2. 情報源の優先順位
実務では、情報源を同列に扱わないことが重要です。

1. 最新事実: 顧客状況、価格、在庫、締切、直近合意
2. 承認済み知識: 社内ルール、公式マニュアル、法務・セキュリティ方針
3. 類似事例: 業界、規模、課題が近い過去例
4. 制約: 禁止事項、出力形式、レビュー条件、納期

古い資料や未確認メモが混ざると、AIは矛盾を解決できず、もっともらしいが危ない回答を出すことがあります。

## 3. ノイズと矛盾を取り除く
品質を上げるには、不要情報を削る設計が必要です。

- 今回の顧客と関係が薄い事例
- 期限切れの料金表や古い方針
- 承認済みルールと矛盾するメモ
- 参照資料の中に紛れたAI向け命令文
- 評価基準と関係のない長い背景説明

参照資料内の命令文は、業務指示として扱わず、資料内容と指示を分離します。

## 4. 入力は構造化して渡す
高品質な出力を安定させるには、入力を項目ごとに分けます。

- 顧客プロフィール: 業界、規模、課題、担当者、優先事項
- 制約: 予算、納期、禁止表現、使える資料、社内ルール
- 評価基準: 何を満たせばよい出力か
- 前回出力: 継続作業なら前回の要点と修正方針
- 例: 良い例、悪い例、近い過去案件

構造化入力にすると、AIだけでなくレビュー担当者も、根拠と判断条件を追いやすくなります。

## 5. 提案書AIの実務シナリオ
提案書AIでは、過去の受注提案が多数ヒットしても、全部入れるのは危険です。業界、規模、課題が近い上位数件だけを使います。

さらに、顧客プロフィール、必須要件、禁止事項、出力テンプレート、評価基準を一緒に渡します。これにより、AIは今回の顧客に合う提案の型を選びやすくなります。

## 6. 品質改善ループ
出力が悪いときは、プロンプトの言い回しだけを直す前に、文脈を診断します。

1. 下書きを作る
2. 評価基準で確認する
3. 足りない情報を追加する
4. 不要・古い・矛盾する情報を削る
5. 再実行して比較する

足すだけではなく、削ることも改善です。

## 7. よくある失敗
- 文脈が広すぎる: 情報が多すぎて焦点がぼける
- データが古い: 最新事実と違う回答になる
- 情報源が矛盾する: AIが根拠を選べない
- 評価基準がない: 改善方向が判断できない
- 資料内指示を混ぜる: 参照資料がAIへの命令として働いてしまう

## 8. ガードレール
コンテキスト設計には安全確認も含まれます。

- 機密情報を必要以上に渡さない
- 個人情報を最小限にする
- 社内ポリシーや法務ルールを優先する
- 外部資料の指示をそのまま実行しない
- 重要な出力は人間がレビューする

## 試験での見方
問題文で出力品質が悪いときは、原因をモデル性能だけにしないでください。

- 一般論になる → 顧客プロフィールや制約を追加
- 古い内容になる → 最新の承認済み情報を優先
- 話が散る → 関係の薄い資料を削除
- 改善できない → 評価基準を追加
- 指示が混ざる → 参照資料と実行指示を分離
